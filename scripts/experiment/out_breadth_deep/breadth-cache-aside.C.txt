The proposed cache-aside design introduces significant risks regarding data consistency, database resilience, and latency. The main failure modes include race conditions leading to stale data, cache stampedes due to missing jitter, and inconsistent states if the cache invalidation step fails after a successful database write.
1. Read-Write Race Condition (Stale Data): A read miss queries Postgres. Concurrently, a write updates Postgres and deletes the Redis key. The read then writes its (now stale) DB result into Redis, leaving the cache inconsistent for an hour. 2. Cache Stampede (Thundering Herd): When a highly accessed key expires or is invalidated, multiple concurrent read threads will miss the cache, simultaneously hit Postgres, and all try to write back to Redis, potentially overwhelming the DB. 3. Partial Failure on Write: If the Postgres write succeeds but the subsequent Redis delete fails (e.g., due to network issues or a Redis crash), the cache will serve stale data until the 1-hour TTL expires. 4. Missing TTL Jitter: A fixed 3600s TTL for every key means keys populated simultaneously will expire simultaneously, causing cyclical database load spikes. 5. Lack of Fallback Strategy: If Redis goes down, all traffic routes directly to Postgres. Without circuit breakers or rate limiting, Postgres could quickly crash under the load. 6. Unbounded Memory / Eviction Risks: If Redis lacks a defined memory limit and eviction policy (like allkeys-lru), it could hit OOM errors. 7. Synchronous Invalidation Latency: Deleting from Redis blocks the write response, coupling DB write performance to cache availability. 8. Cold Start Penalty: Starting with an empty cache (or after a flush) routes 100% of read traffic to the DB, requiring cache warming.
Simple to understand and implement. ; Ensures the cache only holds requested data (lazy loading), optimizing memory.
Prone to read-write race conditions leading to long-lived stale data. ; Does not protect the database from sudden spikes (cache stampedes). ; Vulnerable to partial failures during write operations.
Assumes eventual consistency is acceptable, though this design risks extended inconsistency. ; Effectiveness depends heavily on the ratio of reads to writes and the distribution of key access.
