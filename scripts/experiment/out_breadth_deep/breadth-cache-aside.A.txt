The proposed cache-aside design introduces critical correctness and reliability risks. It is vulnerable to race conditions that can cache stale data for hours, and its lack of TTL jitter or stampede mitigation can lead to cascading database failures under high concurrency.
1. **Read/Write Race Condition (Stale Data)**: A classic cache-aside race occurs if Thread A experiences a cache miss and reads from Postgres. Concurrently, Thread B updates Postgres and deletes the Redis key. Thread A then finishes its read and writes the stale value into Redis. The stale data remains for the 3600s TTL. 
2. **Partial Failure / Non-Atomicity**: The Postgres update and Redis delete are not atomic. If the database update succeeds but the application crashes or network fails before deleting the Redis key, the cache becomes inconsistent with the database for up to an hour. 
3. **Cache Stampede (Thundering Herd)**: If a highly popular key expires (or is explicitly deleted on write), thousands of concurrent read requests will miss the cache simultaneously. They will all hit Postgres at once, potentially exhausting the connection pool or causing a database outage. 
4. **Synchronous Expiration (No Jitter)**: Using an exact fixed TTL (3600s) means that if a bulk load or a cold cache population occurs, all those keys will expire at the exact same moment an hour later, creating periodic, massive traffic spikes to Postgres. 
5. **Inefficient Uniform TTL**: Treating all data with a uniform 1-hour TTL ignores access and mutation patterns. Highly volatile data may be served stale too often, while static data is evicted unnecessarily. 
6. **Write Amplification to Redis**: During a cache stampede, not only does Postgres get hammered with identical queries, but Redis is subsequently hammered with identical write commands as all threads try to populate the missing key. 
7. **OOM Risks (No Eviction Mentioned)**: If the read-heavy service accesses a working set larger than Redis memory within 1 hour, Redis will hit its maxmemory limit. If eviction policies (e.g., volatile-lru) aren't properly configured, this will cause Redis to reject writes or crash.
Conceptually simple to implement. ; Reduces baseline read load on the database in low-concurrency, low-mutation scenarios.
High probability of persisting stale data under concurrent read/write loads. ; High risk of cascading failures (Postgres overload) due to cache stampedes. ; Lack of resilience against partial failures (network partitions or app crashes).
The severity of these issues depends heavily on the actual traffic volume, key popularity distribution (Zipfian), and concurrency levels.
