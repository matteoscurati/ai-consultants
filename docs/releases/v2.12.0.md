## Highlights

- New `ai-consultants init` subcommand scaffolds the directory with a starter `.env` (copied from `.env.example`, chmod 600) and a sourceable `config.sh`.

## What's New

### Persistent User Config

Pre-v2.12.0 you had two options to persist a setting like "always enable debate":
1. `export ENABLE_DEBATE=true` in `.bashrc` (pollutes every shell, not project-scoped)
2. Source a project-local `.env` before each invocation (manual, easy to forget)

v2.12.0 adds a third, conventional option: drop a `.env` or `config.sh` in `~/.config/ai-consultants/` and it's loaded automatically.

```bash
ai-consultants init                              # one-time scaffold
$EDITOR ~/.config/ai-consultants/.env             # set your defaults
ai-consultants doctor                            # confirms it's loaded
ai-consultants "your question"                   # uses the new defaults
```

### Search Path

The user config directory is resolved with this precedence (first found wins):

| Order | Source |
|-------|--------|
| 1 | `$AI_CONSULTANTS_CONFIG_DIR` |
| 2 | `$XDG_CONFIG_HOME/ai-consultants` |
| 3 | `$HOME/.config/ai-consultants` (default) |

Inside the directory:

| File | Format | Loaded by | Notes |
|------|--------|-----------|-------|
| `.env` | KEY=value | `load_user_config()` | `export` prefix supported, `# comments` skipped, quoted values unquoted, existing env vars NOT overridden |
| `config.sh` | bash | `load_user_config()` | Sourced after `.env`, full bash â€” use `${VAR:-default}` to defer to env |
| `affinity.json` | JSON | `lib/routing.sh` | Picked up if present, supersedes bundled matrix; `AFFINITY_FILE` env var still highest priority |

### Precedence

The full precedence chain (highest wins):

1. **CLI flags** (`--preset`, `--strategy`, â€¦)
2. **Existing env vars** (`export FOO=bar` before invocation)
3. **User config** (`~/.config/ai-consultants/{config.sh,.env}`) â€” **new in v2.12**
4. **`scripts/config.sh` defaults** (the `${VAR:-default}` fallbacks)
5. **Hardcoded defaults** in individual scripts

### `bin/ai-consultants` Dynamic Version

The wrapper used to hardcode `VERSION="2.10.0"` and was never bumped through v2.10.x or v2.11.0. v2.12.0 reads the version from `config.sh::AI_CONSULTANTS_VERSION` (single source of truth). No more drift.

### `references/affinity.json` Search Path

Pre-v2.12: `AFFINITY_FILE` env var or bundled default.
v2.12: `AFFINITY_FILE` env var > `~/.config/ai-consultants/affinity.json` > bundled default.

This lets you keep a per-machine custom routing matrix in the user dir without exporting `AFFINITY_FILE` everywhere.

## Pre-Release Review Findings (and fixes)

Four reviewer agents (general code quality, architecture, silent-failure hunt, test coverage) audited the diff before tagging. All blocking and high-severity findings were addressed in the release:

| Severity | Issue | Fix |
|----------|-------|-----|
| BLOCKING | `load_user_config()` runs NĂ—15-30 per consultation because `config.sh` is sourced transitively by every `query_*.sh` and `lib/common.sh`. A user `config.sh` doing PATH-appends, counters, or log-appends would compound silently. | Added a process-wide idempotency guard `_AI_CONSULTANTS_USER_CONFIG_LOADED`. Single load per shell. |
| HIGH | CR-LF line endings in `.env` (Windows users editing in Notepad) silently corrupted values: `ENABLE_DEBATE=true\r` failed every `[[ "$X" == "true" ]]` comparison. | `_apply_env_file` now strips a trailing `\r` from each line. New test `Test 9: CR-LF stripping`. |
| HIGH | `bin/ai-consultants version` could print garbage like `vAI_CONSULTANTS_VERSION=2.12.0` if the version line was ever unquoted. The `sed` regex returns input verbatim on no-match, and the `\|\| echo "unknown"` fallback never fired. | Added an explicit semver pattern check after the parse; failure produces `unknown`. New test `Test 3: version=unknown on malformed config`. |
| HIGH | `bin/ai-consultants init` followed symlinks. A non-root user could pre-create `~/.config/ai-consultants` as a symlink to `/etc/cron.d`; `init --force` run as root would `chmod 600` write into it. | Init refuses to scaffold when the resolved dir is a symlink. Also pre-flights parent-dir write permission with a friendly error. New test `Test 8: init refuses symlinked dir`. |
| MED | User-dir resolution duplicated 4 times: `user_config.sh::get_user_config_dir`, `routing.sh::_resolve_affinity_path`, `doctor.sh::check_user_config`, `bin/ai-consultants::init`. Phase 4 work would have made it 5. | Single source of truth in `lib/user_config.sh`. The other three sites source it. `routing.sh` does so defensively (works standalone for tests). |
| MED | `affinity.json` placed in user dir was untested. The `_resolve_affinity_path` middle branch (between `AFFINITY_FILE` env and bundled default) had zero coverage. | New test in `test_routing_parity.sh` exercises the user-dir branch and verifies `AFFINITY_FILE` still wins over user dir. |
| MED | Container scenario (`docker run` without `-e HOME=...`) silently computed the broken path `/.config/ai-consultants`. | `get_user_config_dir` returns empty + exit 1 when HOME and XDG both unset; callers degrade gracefully. New test `Test 11: HOME unset fallback`. |
| MED | `bin/ai-consultants init` had zero behavioral test coverage. A regression dropping `chmod 600` would silently leak API keys. | New `scripts/test_bin.sh` with 10 assertions over 8 tests. |
| LOW | Test 4 in `test_user_config.sh` asserted "invalid keys rejected" but couldn't read those keys via parameter expansion (bash syntax error). It only verified the loader didn't crash. | Test now greps `env` directly for the invalid keys. |
| LOW | Test 7 checked exit code but not stderr â€” a silent-warning regression would have passed. | Now captures stderr separately and asserts empty. |

Known coverage deferred to a future release:
- `doctor.sh check_user_config()` failure paths (file present with bad perms, etc.) â€” partially covered by integration runs but no dedicated test
- The `test_*.sh` files are still standalone (not invoked by `test_suite.sh` or any CI). Wiring TBD.

## Breaking Changes

None.

The user config dir is purely additive: nothing loads it implicitly unless you create files there. If `~/.config/ai-consultants/` doesn't exist, behavior is identical to v2.11.0.

`bin/ai-consultants` reading the version from `config.sh` is a bug fix (the hardcoded `2.10.0` was wrong). If you parsed `ai-consultants version` output, the format is unchanged (`ai-consultants v<X.Y.Z>`); only the value changes.

## Upgrade Guide

```bash
# git clone
cd ~/.claude/skills/ai-consultants && git pull

# npx (auto-updates per invocation)
npx ai-consultants@latest version    # should print 2.12.0

# curl | bash
curl -fsSL https://raw.githubusercontent.com/matteoscurati/ai-consultants/main/scripts/install.sh | bash
```

After upgrade, optionally bootstrap the user config dir:

```bash
ai-consultants init
$EDITOR ~/.config/ai-consultants/.env
ai-consultants doctor    # 22/22 checks
```

Verify the upgrade:

```bash
ai-consultants version           # 2.12.0 (no longer 2.10.0)
./scripts/doctor.sh              # 22/22 checks pass
./scripts/test_user_config.sh    # 13 assertions, exit 0
./scripts/test_routing_parity.sh # 144 assertions still green
```

No configuration changes required.
