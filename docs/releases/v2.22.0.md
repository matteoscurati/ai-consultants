# Release v2.22.0

**Date:** 2026-07-17
**Type:** Minor — the public `ai-consultants configure` subcommand
**Previous:** v2.21.1

## Highlights

- **`ai-consultants configure`** is now a public subcommand: detect your installed CLIs and API keys, then write a complete, private, persistent configuration.
- **CLI-first transport selection with API fallback.** A consultant with an installed CLI uses it; a consultant with only an API key falls back to the API instead of being dropped.
- **Your settings survive.** Existing custom values and secrets are preserved; only availability-derived `ENABLE_*` flags are refreshed. Rewrites are backed up and files stay at mode 600.
- **Scriptable or guided.** Repeatable `--set KEY=VALUE` for automation, `--interactive` / `--advanced` for a guided review, `--show-parameters` to discover the contract, and a side-effect-free `--dry-run` that redacts secrets.
- The stale 1,092-line v2.0 setup wizard is gone; `setup_wizard.sh` now forwards to the maintained configurator.

## What's New

### `ai-consultants configure`

Automatic mode is the default. It detects the current 11-consultant roster, resolves each transport, and writes the result to your XDG user config directory:

```bash
ai-consultants configure
```

The accepted parameter surface is derived from `.env.example`, so `--show-parameters` is always an accurate contract rather than a hand-maintained list.

| Flag | Purpose |
|------|---------|
| `--auto`, `--non-interactive` | Configure from detected CLIs and API keys (default) |
| `--interactive` | Review credentials, transports, and panel selection |
| `--advanced` | Review every supported persistent parameter |
| `--set KEY=VALUE` | Override any parameter; repeatable, and participates in detection |
| `--show-parameters` | List every parameter accepted by `--set` |
| `--dry-run` | Print the generated config without writing it (secrets redacted) |
| `--output FILE` | Write to `FILE` instead of the user config `.env` |
| `--force` | Overwrite without creating a backup |

### Transport selection

The rule is CLI-first: if a consultant's CLI is installed, it is used. API mode is a fallback for a consultant that has a key but no CLI, or an explicit choice you make.

Auto-selected transports are written with an `# ai-consultants:auto` provenance marker, so a later run can adapt when a CLI appears or a key is added — without ever overriding a transport you pinned yourself. An unmarked value is treated as your deliberate pin and is left alone.

**The starter template's transport switches are commented out on purpose.** `CODEX_USE_API`, `CLAUDE_USE_API`, `MISTRAL_USE_API`, and `QWEN3_USE_API` ship commented in `.env.example` — that is what keeps auto-resolution in charge. Uncomment one only when you want to force that consultant's transport.

## Breaking Changes

None. Every change is backwards-compatible.

`setup_wizard.sh` still runs, but forwards to `configure` — it no longer advertises removed consultants or performs incomplete authentication checks.

## Upgrade Guide

```bash
# If installed via npm/npx
npx ai-consultants@latest configure

# If installed via git clone
cd ~/.claude/skills/ai-consultants && git pull

# If installed via curl | bash
curl -fsSL https://raw.githubusercontent.com/matteoscurati/ai-consultants/main/scripts/install.sh | bash
```

No configuration changes are required. If you already have a config, `configure` preserves your custom values and secrets and backs the file up before rewriting it.

If you previously ran `init` and found consultants unexpectedly disabled, run `configure` again after upgrading — see below.

## Fixed

- `configure --interactive` no longer exits without saving the moment you press Enter to keep a value.
- `configure --advanced` no longer writes wrong values: each prompt was reading the next parameter's name instead of your answer.
- `init` followed by `configure` no longer disables Codex, Claude, Mistral, and Qwen3 when only an API key is available. The starter template's transport switches are now commented out, so auto-detection stays in charge until you deliberately set one. An installed CLI still takes precedence over an API key.
- Configuration backups can no longer overwrite each other when two runs land in the same second.
- The npm package now includes the full `.env.example`, so `init` no longer falls back to a partial five-setting template.
- Gemini API setup consistently uses the runtime-supported `GEMINI_API_KEY`; `.env` inline comments are parsed correctly, and the starter template no longer pins obsolete `/tmp` storage paths.
- Repeatable `--set` values participate in transport detection with final precedence, including intentional empty values.
- Existing `export KEY=value` settings survive rewrites, `.env.example` is safely sourceable, and `--dry-run` no longer creates the persistent config directory.
