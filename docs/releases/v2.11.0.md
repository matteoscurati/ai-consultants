# Release v2.11.0

**Date:** 2026-05-03
**Type:** Minor — Externalized affinity matrix, JSON-based routing, regression tests
**Previous:** v2.10.9

## Highlights

- Routing affinity matrix lives in `references/affinity.json` instead of `lib/routing.sh` — ~190 lines of nested bash `case` statements replaced with a 60-line JSON file you can edit
- `AFFINITY_FILE=/path/to/custom.json` lets you override the matrix per consultation without touching the codebase
- `doctor.sh` validates the JSON schema and coverage (any consultant missing from a category is flagged) — total checks: 21
- New golden test (`scripts/test_routing_parity.sh`) — 144 assertions covering all 9 × 14 = 126 (category, consultant) scores plus edge cases (cache invalidation, cache substring collision, AFFINITY_FILE override)
- New regression test (`scripts/test_set_e_safety.sh`) — static lint of the `((var++))` family (`++`, `--`, `let`) plus dynamic semantics check on bash 4+
- Pre-release multi-agent review caught and fixed: cache substring collision in `get_affinity()`, `consult_all.sh` swallowing doctor's output (silent-failure regression), `((attempt++)) || true || true` artifacts in `api.sh`, lint regex too narrow

## What's New

### External Affinity Matrix

Before v2.11.0, `scripts/lib/routing.sh` contained a 190-line `get_affinity()` function with 9 nested `case` statements — one per question category. Adding a new consultant required editing 9 places. Tweaking scores for a specific project required forking the script.

Now: `references/affinity.json` is the single source of truth.

```json
{
  "version": "1.0",
  "default_score": 5,
  "general_score": 8,
  "known_consultants": ["Gemini", "Codex", "..."],
  "categories": {
    "CODE_REVIEW": { "Gemini": 7, "Codex": 10, "Mistral": 8, "..." },
    "SECURITY":    { "..." }
  }
}
```

### Per-Consultation Customization

Override the matrix at runtime without editing the codebase:

```bash
# Example: a project where DeepSeek's algorithm specialism matters more
AFFINITY_FILE=~/projects/algo-heavy/affinity.json \
  ./scripts/consult_all.sh "redesign the priority queue"
```

Use cases:
- Project-specific weighting (favor security-focused consultants for an auth refactor)
- Excluding consultants from a category by setting their score below `MIN_AFFINITY` (default 7)
- A/B testing alternative matrices before upstreaming
- Disabling smart routing entirely by pointing at a flat all-8 matrix

The lookup logic is:

1. If the consultant is not in `known_consultants` → `default_score` (5)
2. Else if the category is not in `categories` (e.g. `GENERAL` or unrecognized) → `general_score` (8)
3. Else → `categories[category][consultant]`, falling back to `default_score` (5) if missing

This preserves the exact pre-v2.11.0 behavior; the parity test below proves it.

### Two-Level Caching

`get_affinity()` is called ~14 times per consultation (once per consultant). Naive jq invocation per call would add ~40-400ms per consultation depending on the system. v2.11.0 caches at two levels:

- **File-content cache**: the JSON is read once per shell, kept in memory; reset only if `AFFINITY_FILE` changes
- **Result cache**: each (category, consultant) pair is cached after first lookup, so subsequent calls are O(1) string lookups

After warm-up, ~13 of 14 calls per consultation are pure-bash cache hits. Total overhead: <100ms per consultation.

### Doctor Validation (3 new checks → 21/21)

`./scripts/doctor.sh` now verifies the affinity matrix:

```
▶ Checking Routing Affinity Matrix (v2.11)
────────────────────────────────────────────────────────────────
  ✓ Affinity file: references/affinity.json
  ✓ Affinity schema: valid (default_score, general_score, known_consultants, categories)
  ✓ Affinity coverage: 14 consultants x 9 categories
```

Coverage check catches the failure mode that motivated this release: when v2.8 added Amp, v2.9 added Kimi, and v2.10 added MiniMax, each new consultant had to be inserted into 9 separate `case` blocks. Easy to forget one. Now `doctor.sh` flags missing entries explicitly.

### Golden Parity Test

`scripts/test_routing_parity.sh` runs 139 assertions:

- **126 golden values** — every (category, consultant) score, locked in as expected literals
- **4 unknown-consultant** edge cases (always returns 5)
- **6 unknown-category** edge cases (returns 8 for known consultants)
- **3 AFFINITY_FILE override** cases (custom JSON is picked up correctly)

Any future change to the JSON requires updating the test on purpose, not by accident.

### `set -e` Safety Regression Test

`scripts/test_set_e_safety.sh` codifies the lesson from v2.8.1, v2.10.1, and v2.10.9: bare `((var++))` aborts under `set -e` on bash 4+. The test does:

1. **Static lint** — greps every `.sh` file for unprotected `((var++))` and fails if any exist (excluding the deprecated `preflight_check.sh` wrapper)
2. **Dynamic verification** — if bash 4+ is on the system, runs synthetic snippets to confirm the abort/no-abort semantics still hold (skipped on bash 3.2 macOS where the bug doesn't manifest)

Run as part of CI to prevent v2.10.10 from re-introducing the bug.

### Documentation

`docs/SMART_ROUTING.md` was substantially rewritten. The big per-consultant score table (lines 22-33 in v2.10) had drifted from the code since v2.8 — Amp, Kimi, and MiniMax columns were missing or wrong, and several values for older consultants no longer matched. Rather than re-sync the duplicate, the doc now points to `references/affinity.json` as the source of truth and documents the schema + override pattern.

## Pre-Release Review Findings (and fixes)

Before tagging, four reviewer agents (general code quality, architecture, silent-failure hunt, test-coverage analysis) audited the diff. They surfaced six issues; all were fixed in the release:

| Severity | Issue | Fix |
|----------|-------|-----|
| HIGH | `get_affinity()` cache used `*"${key}"*` glob match, where `key="CAT|C="`. Lookup of `DEBUG|Codex=` would substring-match a cached `BUG_DEBUG|Codex=10` and return the wrong score. Dormant on shipped JSON, exploitable via `AFFINITY_FILE` or future categories. | Cache now uses leading-space delimiter (`" CAT|C="`); lookup pattern is `* CAT|C=*`. Two collision regression tests added to parity suite. |
| HIGH | `consult_all.sh:222` invoked `doctor.sh > /dev/null 2>&1` — repeating the exact silent-failure pattern that motivated v2.10.9. User saw "Pre-flight check failed" with no detail. | Output is now captured to a tmpfile and dumped to stderr on failure; success path stays quiet. |
| MED | `((attempt++)) || true || true` on three lines in `lib/api.sh` (the v2.10.9 mechanical sweep applied `|| true` on top of pre-existing protections). | Cleaned to single `|| true`. |
| MED | `test_set_e_safety.sh` lint regex caught only `((var++))`, missed `((var--))`, `((var+=1))`, `let var++`. | Regex broadened to cover `++` and `--` post-ops + `let`-form. Pre-ops `((++var))` documented as safe (initial 0 → new value 1 → exit 0). |
| MED | The 3-tier fallback (unknown consultant → 5, unknown category → 8, missing cell → 5) is asymmetric and was undocumented. | `_comment` in `affinity.json` now spells out the rationale (missing cell = config gap, should de-rank; unknown category = unmapped routing, treat known consultants equally). |
| MED | Cache auto-invalidation when `AFFINITY_FILE` changes mid-session was not properly tested — the existing test manually reset `_AFFINITY_LOADED_FILE`, so a regression in the auto-invalidation logic would have been silent. | Test rewritten to rely on the auto-invalidation path; warm cache against default, then switch to override without manual reset. |

Known coverage gaps deferred to a future release (not blocking v2.11.0):
- `preflight_check.sh` wrapper has no behavioral test (forwarding, exit-code propagation, TTY-gated warning)
- `doctor.sh check_routing()` failure paths (missing file, malformed JSON, schema gaps, coverage gaps) not driven by tests
- `test_routing_parity.sh` and `test_set_e_safety.sh` are not invoked from `test_suite.sh` — they must be run manually or wired into CI

## Breaking Changes

None.

The migration is purely internal: `get_affinity(category, consultant)` returns the same scores as before. The only observable change is positive — `AFFINITY_FILE` is now meaningful where it did nothing before.

If you had previously edited `lib/routing.sh` with custom scores, port them to `references/affinity.json` (or to a custom file pointed at by `AFFINITY_FILE`).

```bash
# git clone install
cd ~/.claude/skills/ai-consultants && git pull

# npx (auto-updates per invocation)
npx ai-consultants@latest doctor

# curl | bash
curl -fsSL https://raw.githubusercontent.com/matteoscurati/ai-consultants/main/scripts/install.sh | bash
```

Verify the upgrade:

```bash
./scripts/doctor.sh                    # should report 21/21 checks
./scripts/test_routing_parity.sh       # 139 assertions, exit 0
./scripts/test_set_e_safety.sh         # static + dynamic, exit 0
```

No configuration changes required.
