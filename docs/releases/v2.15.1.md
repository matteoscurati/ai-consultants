# Release v2.15.1

**Date:** 2026-06-22
**Type:** Patch — Gemini works out-of-the-box for npm/npx users
**Previous:** v2.15.0

## Highlights

- **Gemini responses are no longer degraded.** v2.15.0's default Gemini model returned its JSON inside a markdown fence that the parser didn't handle — every reply collapsed to a generic fallback. Now parsed correctly (summary, approach, pros/cons, confidence all populated).
- **Gemini no longer silently disappears on a fresh `npx` install.** The transport (agy CLI vs Google AI API) is now auto-resolved from your environment.
- Just export `GEMINI_API_KEY` — Gemini switches to API mode automatically. No need to also set `GEMINI_USE_API=true`.
- `doctor` stops reporting a missing `agy` CLI as a failure when Gemini is running in API mode, and points you at the API path when the CLI is absent.
- Fully backwards-compatible: agy/OAuth users keep CLI mode, and any explicit `GEMINI_USE_API` is honored.

## What's New

### Markdown-fenced JSON now parses (degraded Gemini responses)

v2.15.0 assumed the agy CLI prints the model's JSON envelope directly. Verified against agy 1.0.10, that holds only for some models: `Gemini 3.5 Flash (Low)` returns bare JSON, but the **default** `Gemini 3.1 Pro (High)` wraps it in a ```` ```json … ``` ```` fence. The parser's `.response.summary` check failed on the fenced text, so every Gemini reply fell through to the fallback path — `summary: "Unstructured response - see detailed"`, `confidence: 5`, empty pros/cons, with the real envelope stranded as a fenced string inside `.detailed`.

A shared `strip_json_fence` helper now removes the fence when the text isn't already valid JSON. It's a fallback-only step, so bare-JSON models (Flash, and every other consultant) are untouched, and it's consultant-agnostic — any CLI model that fences its JSON now parses cleanly. The helper is applied on every path that consumes Gemini output — the main response processor, self-reflection/refinement, and synthesis (peer review already had its own fence handler).

### Synthesis via agy no longer hangs

`build_synthesis_args` invoked agy without a `--print`/`-p` flag (`echo "$prompt" | agy`), so — unlike codex (`--full-auto`) and claude (`--print`) — agy launched its interactive session and never consumed the piped prompt. Whenever agy was the chosen synthesizer (e.g. Claude Code is the invoking agent and agy is on PATH), synthesis produced nothing. Now invoked as `agy -p - --model …`, matching the consultant path.

## What's New

### The problem (v2.15.0 regression for npm users)

v2.15.0 migrated the Gemini consultant to Google's Antigravity CLI (`agy`). But Gemini ships **enabled by default** in **CLI mode**, and `agy`:

- **can't be installed via npm** — it's a `curl|bash` binary that lands in `~/.local/bin/agy`, so a fresh `npx ai-consultants` user almost never has it, and
- is **OAuth-only** — even when installed it requires an interactive browser sign-in, with no headless/API-key path.

The result: every fresh `npx ai-consultants "…"` run silently dropped Gemini from the panel — **even when `GEMINI_API_KEY` was already exported** — because API mode was opt-in and nothing auto-detected the situation. The API path (plain `curl` + key, no binary, no browser) is the npm-friendly one, but it was off by default.

### The fix: transport auto-resolution

When `GEMINI_USE_API` is **unset**, the mode is now resolved from your environment:

| Environment | Resolved mode |
|---|---|
| `GEMINI_API_KEY` is set | **API** (the npm-friendly path — pure HTTP) |
| no key | **CLI** (`agy`; gracefully dropped if absent, with a `doctor` hint) |

An explicit `GEMINI_USE_API=true` or `GEMINI_USE_API=false` is **always** honored, so nothing changes for users who pinned it. The resolution is idempotent across the many `config.sh` re-sources per consultation.

Only Gemini auto-resolves — the other four switchable agents (Codex, Claude, Mistral, Qwen3) keep their `false` defaults, since their CLIs are installable via npm/pip.

### Better `doctor` diagnostics

- A consultant running in API mode no longer fails the CLI install check — previously a missing CLI was a hard `✗ NOT INSTALLED` regardless of mode (this also fixes the same latent false-positive for Codex/Claude/Mistral/Qwen3 in API mode).
- When Gemini *is* in CLI mode and `agy` is missing, the error now adds: `tip: set GEMINI_API_KEY to use API mode (no CLI install needed)`.

## Breaking Changes

None. The only behavioral change is that a key-only environment now reaches Gemini over the API instead of silently dropping it.

## Upgrade Guide

```bash
# If installed via git clone
cd ~/.claude/skills/ai-consultants && git pull

# If installed via curl | bash
curl -fsSL https://raw.githubusercontent.com/matteoscurati/ai-consultants/main/scripts/install.sh | bash
```

No configuration changes required. To use Gemini over the API, just `export GEMINI_API_KEY=…` — you no longer need to also set `GEMINI_USE_API=true`. To keep using the agy CLI, install it (`curl -fsSL https://antigravity.google/cli/install.sh | bash`) and sign in once with `agy`.
