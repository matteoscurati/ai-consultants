# Release v2.19.1

**Date:** 2026-06-27
**Type:** Patch — code-review cleanup for v2.19.0
**Previous:** v2.19.0

## Highlights

- **Fixes the "Diagnosed Failures" report table breaking** when a consultant's error reason contains a `|`.
- **The opt-in health gate no longer wastes a ping** on consultants whose answer is already cached.

## What's New

### Report table escapes pipes

A failure reason containing a `|` (e.g. a CLI error that mentions a piped command) used to mangle the markdown row in the **Diagnosed Failures** table. The reason is now escaped (`|` → `\|`) in table mode. (This was a rendering bug, not a security issue — the text is your own CLI's stderr.)

### Cache-aware health gate

When `ENABLE_HEALTH_GATE=true` and semantic caching is on, a consultant whose response is already cached is kept **without** a ping — Round 1 would serve it for free, so probing it would defeat the cache. The gate only pings consultants it would actually run live.

### Internal cleanup (from `/code-review max`)

- A single `render_diagnosed_failure` helper now renders both the console log and the report table — the `name|reason` decode lives in one place.
- `ping_consultant` takes the already-lowercased consultant id (callers compute it anyway), removing a redundant `to_lower` fork per consultant.
- The health gate's inherent pre-run startup latency (it's serial with the consultation by design, up to `HEALTH_GATE_TIMEOUT`) is now documented in `config.sh`.
