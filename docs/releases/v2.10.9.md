# Release v2.10.9

**Date:** 2026-05-03
**Type:** Patch — Preflight bug fix + deprecation, defensive `set -e` sweep, shellcheck cleanup
**Previous:** v2.10.8

## Highlights

- Fixes a silent failure in `preflight_check.sh`: helper `check_cli_installed` returned non-zero on missing CLIs, and call sites lacked `|| true` — `set -euo pipefail` aborted the script after the "Checking CLI installations..." banner, with no diagnostic output to the user
- Deprecates `preflight_check.sh` (still v2.0 banner, only 6/15 consultants covered, missing CLI/API mode and Ollama checks) in favor of `doctor.sh`
- Adds `--suggest-config` and `--quick` flags to `doctor.sh` for backward-compat parity, with `--suggest-config` now covering all 15 consultants (CLI + API key detection)
- Defensive `((var++)) || true` sweep across 7 files (15 sites) — protects against `set -e` aborts on bash 4+, matching the convention introduced in v2.8.1 and v2.10.1
- Real shellcheck warnings addressed in 4 files (SC2059 unsafe printf, SC2012 `ls | wc -l`, plus 2 annotated false positives)

## What's New

### Preflight Bug Diagnosis

Running `./scripts/preflight_check.sh` on a system missing any consultant CLI exited with code 1 immediately after printing "Checking CLI installations..." — no progress, no error message, no list of installed/missing CLIs. The user had no way to know which CLI was missing.

Root cause: `preflight_check.sh` runs under `set -euo pipefail`. The `check_cli_installed` function returns `1` when the CLI is missing (intentionally, so the caller knows). But the call sites at lines 162-167:

```bash
check_cli_installed "Gemini" "$GEMINI_CMD" "..."
check_cli_installed "Codex"  "$CODEX_CMD"  "..."
check_cli_installed "Mistral" "$MISTRAL_CMD" "..."
# ...
```

…lacked `|| true`. The first missing CLI triggered `set -e`, aborting the script. Only `suggest_configuration()` had `|| true` after each call, which is why `--suggest-config` worked while the default mode did not.

### Deprecation in Favor of `doctor.sh`

Rather than fix the bug in place, `preflight_check.sh` is now a thin wrapper that delegates to `doctor.sh`:

```bash
$ ./scripts/preflight_check.sh --quick
WARNING: preflight_check.sh is deprecated since v2.10.9. Use doctor.sh instead.
         Forwarding to: doctor.sh --quick
[doctor output follows...]
```

`doctor.sh` is the strict superset:
- Covers all 15 consultants (preflight covered 6)
- Reports CLI/API mode (preflight predates v2.6 mode switching)
- Checks Ollama, synthesis engine, configuration, dependencies (jq, bc, claude)
- Produces structured fix suggestions
- 18 checks vs preflight's 6

The deprecation warning prints only when stderr is a TTY, so the wrapper stays silent in scripted contexts (e.g., the `ENABLE_PREFLIGHT=true` path in `consult_all.sh`).

### `doctor.sh --suggest-config` (Ported from preflight)

Now covers 11 CLI consultants (Gemini, Codex, Mistral, Kilo, Cursor, Aider, Amp, Kimi, Claude, Qwen3, Ollama) plus 4 API-only consultants (GLM, Grok, DeepSeek, MiniMax) detected via API key environment variables:

```
$ ./scripts/doctor.sh --suggest-config
# Recommended configuration based on detected CLIs and API keys
# Generated by: ./scripts/doctor.sh --suggest-config
# Available consultants: 10/11 CLI + API

ENABLE_GEMINI=true
ENABLE_CODEX=true
...
```

### Defensive `set -e` Sweep

The codebase already uses `((var++)) || true` consistently (20+ existing sites, established in v2.8.1 and v2.10.1 after similar bugs). v2.10.9 audits the remaining gap:

| File | Sites protected |
|------|-----------------|
| `peer_review.sh` | 4 (`index`, `response_count`, `success_count`) |
| `setup_wizard.sh` | 1 (`AVAILABLE_COUNT`) |
| `test_functions.sh` | 2 (`passed`, `failed`) |
| `lib/api.sh` | 5 (`attempt` retry counter) |
| `lib/common.sh` | 3 (`attempt`, `response_count`) |
| `lib/reflection.sh` | 2 (`overconfidence_count`, `total_count`) |
| `doctor.sh` | 1 (`enabled_count`) |

On bash 3.2 (macOS default), bare `((x++))` does NOT abort under `set -e` — which is why these worked in practice. On bash 4+ (Linux CI, Homebrew bash), the same expression returns exit 1 when `x` was 0 and aborts. Fixing now protects users on those environments.

### Shellcheck Cleanup

Real warnings (not stylistic) addressed:

| File:line | Warning | Fix |
|-----------|---------|-----|
| `lib/progress.sh:158,167` | SC2059 unsafe printf format | Switched to `printf '\033[%sA' "$num_lines"` |
| `install.sh:264` | SC2012 `ls | wc -l` race | Replaced with `find ... -type f | wc -l` |
| `query_ollama.sh:23` | SC2329 unused function | Annotated (invoked via `trap cleanup EXIT`) |
| `peer_review.sh:174` | SC2016 sed in single quotes | Annotated (literal pattern is intended) |

Stylistic warnings (SC2001, SC2004, SC2126, SC2321) are tracked but deferred — they don't affect correctness.

## Breaking Changes

None. `preflight_check.sh` continues to work; it just prints a deprecation notice on TTY and forwards to `doctor.sh`. All flags are mapped 1:1.

## Upgrade Guide

```bash
# If installed via git clone
cd ~/.claude/skills/ai-consultants && git pull

# If installed via npx (no action needed — auto-updates per invocation)
npx ai-consultants@latest doctor

# If installed via curl | bash
curl -fsSL https://raw.githubusercontent.com/matteoscurati/ai-consultants/main/scripts/install.sh | bash
```

If your scripts or CI directly invoke `./scripts/preflight_check.sh`, consider migrating to `./scripts/doctor.sh` — same flags, broader coverage. The wrapper will continue to work for the foreseeable future, but `preflight_check.sh` will be removed in a future major release.

No configuration changes required.
