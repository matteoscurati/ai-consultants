# Release v2.20.0

**Date:** 2026-07-12
**Type:** Minor — capability-aware deliberation + measured roster calibration
**Previous:** v2.19.2

## Highlights

- **Capability-aware voting & composition** — weight votes and rank the panel by each model's capability on the quality axis a question stresses (intelligence or taste). Opt-in.
- **Roster audit** — see which consultants add distinct value vs merely echo the panel.
- **Measured calibration** — derive the capability scores from data (blind peer review + observed cost + pairwise-judge Elo) instead of heuristics.
- Everything new is opt-in and additive: with the flags off, behavior is identical to v2.19.2.

## What's New

### Capability axes (`affinity.json` v1.1)

Beyond category *fit* (the affinity matrix), each consultant now has a `capabilities` score on three axes (1-10) and a `category_axis` map names the quality axis each category stresses — **taste** for API_DESIGN / ARCHITECTURE / CODE_REVIEW / GENERAL, **intelligence** otherwise.

| Flag (default off) | Effect |
|---|---|
| `ENABLE_CAPABILITY_WEIGHTING` | A consultant's vote weight becomes `confidence × (S + capability) / S` on the run's axis (`S = CAPABILITY_WEIGHT_STRENGTH`, default 10). |
| `ENABLE_CAPABILITY_ROUTING` | Panel *eligibility* still uses raw affinity, but eligible consultants are ranked by `affinity + capability`, so under a size limit the quality axis reorders who makes the cut. |

`cost` is a composition/budget axis only — never a vote weight (tie-break order: intelligence > taste > cost).

### Roster audit — uncorrelated value

`scripts/roster_audit.sh` scores each consultant's *distinctiveness* across past consultations: an approach with low keyword overlap versus every other consultant that round is "distinct". A consultant rarely distinct is correlated with the panel (candidate to drop or down-weight); one often distinct earns its seat. Also available as `doctor.sh --roster-audit` and the `/ai-consultants:roster-audit` slash command. Read-only.

### Measured calibration (Tier A + Tier B)

Replace the heuristic capability seeds with measurements, each derived from a signal the tool already records and sliced by the same `category_axis`:

- **`scripts/roster_calibrate.sh` (Tier A)** — intelligence/taste = mean **blind peer-review** score on that axis's categories; cost = mean **observed** `tokens_used` × catalog rate (60/40 split, rank-normalized, cheapest = 10 → cost per *task*, not per token).
- **`scripts/taste_elo.sh` (Tier B)** — taste via **pairwise LLM-as-judge Elo** on taste-axis answers (removes the peer panel's self-bias). Judge is pluggable: `JUDGE_CLI` (default `claude`) or `TASTE_JUDGE_CMD`.
- **`scripts/run_calibration.sh` + `references/calibration_benchmark.json`** — 50 questions (20 taste / 30 intelligence) run through the panel with `ENABLE_PEER_REVIEW=true`, then calibrated.

Both calibrators accept `--json` and `--write` (merge measured scores into `affinity.json`, backing it up first).

## Breaking Changes

None. All features are opt-in; with the flags off, behavior is identical to v2.19.2. The shipped `capabilities` scores are subjective seeds (Claude/Codex grounded on the model-routing table, the rest heuristic) until you run the calibration workflow.

## Upgrade Guide

```bash
cd ~/.claude/skills/ai-consultants && git pull   # or: npx ai-consultants@latest install
```

Optional — measure capability scores for your own panel:

```bash
./scripts/run_calibration.sh              # collect data (real calls; --limit to sample)
./scripts/roster_calibrate.sh --write     # Tier A: intelligence/taste/cost -> affinity.json
./scripts/taste_elo.sh --write            # Tier B: refine taste (optional)
```
