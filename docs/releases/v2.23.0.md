# Release v2.23.0

**Date:** 2026-07-20
**Type:** Minor — removes a dead feature flag and repairs the release gate
**Previous:** v2.22.0

## Highlights

- **`ENABLE_REFLECTION` / `REFLECTION_CYCLES` removed.** The self-reflection module they controlled was never called by anything, so the settings had never affected a consultation.
- **The release gate works again.** `scripts/release.sh` could not pass its own `npm test` since 2.22.0; two test suites failed deterministically against any real user configuration.
- **`npm test` now reports failures it used to swallow.** A failing assertion followed by a passing one inside the same test function was counted as a pass.
- **A registry that could enroll feature flags as consultants is fixed**, and pinned by a test so it cannot drift again.
- **Preset documentation corrected** — the tables described a deliberation depth the code has not enforced since v2.16.

## What's New

### Self-reflection is gone

`lib/reflection.sh` implemented a generate-critique-refine loop across 452 lines. Nothing ever sourced it. `ENABLE_REFLECTION` and `REFLECTION_CYCLES` were declared in `config.sh` and set by four presets, but no consumer read them — the flags were configuration that promised behavior the tool never performed.

The module had also aged past the codebase. Its `_exec_consultant` hardcoded a three-branch dispatch (Gemini/Codex/Mistral) predating both the 11-consultant roster and the CLI/API transport layer, and its Gemini branch still used the CLI syntax retired in v2.15.0 by the `agy` migration. Wiring it up as written would have produced reflection for 3 of 11 consultants with a broken Gemini path.

It was removed rather than implemented because v2.16 already covers the ground: the `converge` and `adversarial` orchestration shapes run critique-refine driven by measured consensus instead of a fixed cycle count.

### The release gate

`scripts/release.sh` sources `lib/common.sh` — and so `config.sh`, and so `load_user_config` — before running `npm test`. Every setting in the maintainer's own `~/.config/ai-consultants/.env` was therefore exported into the test process, and two suites were not isolated against it:

- **`test_user_config.sh`** — `test_xdg_fallback` unsets `AI_CONSULTANTS_CONFIG_DIR` in the suite's own shell, which it must in order to assert the XDG branch, but nothing restored it. The unset leaked into all 22 following tests, which then read the real user config. Two further tests asserted on real configuration variable names that an ambient export silently replaced.
- **`test_configure.sh`** — its `run_clean_configure` helper scrubbed the nine API keys but no settings, while `configure`'s `has_explicit_value` reads *any* ambient value as a deliberate user pin. An exported `DEFAULT_PRESET` overrode the value the preservation tests had just written into the file they were asserting on.

Both suites now derive their isolation from the contract they test rather than from a hand-listed subset: `test_user_config.sh` re-establishes an empty config directory before every test, and `test_configure.sh` builds its scrub set from `configure --show-parameters`.

This is why the failure was invisible: `configure` shipped in 2.22.0, so only a maintainer who had actually run it hit the condition, and the first failing suite aborted the run before the second could be seen.

### Test failures that were being swallowed

`test_functions.sh` scores a test function by its exit status, which in bash is that of its last command. A failing assertion followed by a passing one printed `FAIL`, counted as a pass, left the failure total at zero and exited 0. Eight of the thirteen test functions run more than one assertion and were exposed.

The `assert_*` helpers now record failures in a flag the runner consults, so a test fails if either its status or any assertion inside it did.

### Phantom consultants from feature flags

`_discover_custom_api_agents` walks the environment and enrolls any `ENABLE_X=true` as a consultant when an `X_API_URL` is also present and `X` is absent from `KNOWN_FEATURE_FLAGS`. That registry listed 9 entries against 27 live non-consultant flags, so 18 — `ENABLE_PEER_REVIEW` and `ENABLE_HEALTH_GATE` among them — were latent phantom consultants. All 27 are registered, and a test derives the expected set from `config.sh` so the next added flag fails the build instead of drifting silently.

### Preset documentation

The tier-preset tables stated a per-preset debate and peer-review depth that the code does not enforce. Under the default `ORCHESTRATION_MODE=auto` the planner chooses the shape per question: a `DEBATE_ROUNDS` value a preset pins applies only under `ORCHESTRATION_MODE=fixed`, and a `SECURITY`-classified question runs the adversarial shape — peer review included — whatever preset was chosen. `max_quality` was also described as "all consultants" while enabling 8 of the 11-consultant roster.

## Breaking Changes

**`ENABLE_REFLECTION` and `REFLECTION_CYCLES` are no longer accepted parameters.**

No runtime behavior changes — the flags controlled nothing. What changes is the configuration surface:

- `ai-consultants configure --set ENABLE_REFLECTION=...` now exits non-zero instead of accepting the value. A provisioning script that pins either key will abort after upgrading.
- Both keys are dropped, without warning, the next time `configure` rewrites an existing user `.env`.

## Upgrade Guide

```bash
# If installed via git clone
cd ~/.claude/skills/ai-consultants && git pull

# If installed via curl | bash
curl -fsSL https://raw.githubusercontent.com/matteoscurati/ai-consultants/main/scripts/install.sh | bash

# If installed via npx — nothing to do, npx resolves the published version
```

Remove `ENABLE_REFLECTION` and `REFLECTION_CYCLES` from any `.env` file or provisioning script that sets them. No other configuration changes are required.
